Living Documents · format direction

Clean file + lock file

Keep the document as raw, hand-editable Markdown you can open in Obsidian, diff in git, and hand to an agent — while it still carries bound values, provenance, and a review trail. The trick is borrowed from package managers: a clean file you write, and a generated lock file that remembers where every number came from.

1The mental model

Two files per document. You own one; the app maintains the other.

Weekly Summary.md

The document. ~99% pure Markdown prose that you write and edit anywhere. The only departure: a small token {{metrics.mrr}} wherever a value is pulled from a source.

+

Weekly Summary.lock.json

Hidden. Generated & maintained by the app. Remembers each binding's resolved value, its source, a hash of the source at last sync, and who applied it. This is the provenance ledger.

Like package.json + package-lock.json — you hand-edit the manifest; the lock file is the machine's record of exactly what got resolved. Delete the lock and the app rebuilds it from the sources.

2What's actually on disk

The file stays legible. Bound values are visible, meaningful tokens — never smuggled metadata.

      "sourceHash": "a1b2c3",
      "syncedAt":   "2026-06-21T09:02Z",
      "kind":       "figure"
    }
  },
  "context": {            // influence edges
    "market-research.md": {
      "reviewedHash": "9f0e1d",
      "reviewedAt":   "2026-06-19T…",
      "scope":        "document"
    }
  },
  "claims": {
    "commentary-tone": {
      "anchor":  "Growth accelerated…",
      "boundTo": ["metrics.mrr"],
      "kind":    "meaning",
      "state":   "applied"
    }
  }
}

The token is the anchor. Reorder paragraphs, insert ten lines above it, paste it elsewhere — {{metrics.mrr}} still resolves. No line numbers to drift, nothing to silently break. (Contrast: anchoring by line number breaks the instant anyone adds a line above.)

4When meaning changes, not just numbers

A token covers a value. But a whole paragraph's sentiment can be informed by another document — and that's the real superpower.

exactValue binding

{{metrics.mrr}} → a cell. 1:1, deterministic, a hash compare tells you it changed. Cheap.

  • "Revenue grew to $48.6k"

judgmentInfluence dependency

A source shapes the framing or facts of prose — one-to-many, diffuse. "Did it change in a way that matters, and where?" is a model's judgment, not a hash.

  • A shifting product principle rewording a thesis
  • A new competitor that should ripple through every mention

The architecture: generalize the lock file's binding table into a dependency graph, then split cheap staleness detection (always-on, a dirty bit — costs nothing) from expensive impact computation (a model call, only on demand). Impact results become pending changes in the same review rail — no new grammar.

Market Landscape

The strategy rests on truth-maintenance becoming the scarce layer as generation gets cheap.

Today there are no direct competitors shipping a review-first document engine.

Our wedge is provenance: every claim traces to a source, every edit was approved.

Try it: market-research.md changed (a new competitor appeared). Click Review impact — a model pass reads the diff against the whole doc and proposes candidates, routed by confidence: a high-confidence factual add, and a lower-confidence sentiment softening. Both land in the review rail for your call. Nothing was eagerly rewritten.

6Multiplayer, without painting us in a corner

Keep the provenance layer keyed by binding identity, never by text position. Then collaboration is a layer we can upgrade — not an architecture we have to redo.
⌨️

Text layer — how bytes merge

Start with save-based / file-level editing. Swap in a CRDT (Yjs / Automerge) later for Google-Docs-grade simultaneous typing. Markdown is just its serialization.

upgrade later
🔗

Binding & provenance layer — the lock file

Keyed by identity (metrics.mrr, claim IDs), not by line/character position. So it rides on top of any text layer, untouched, whether bytes merge via save or via CRDT.

build now

The one commitment we make today: identity-keyed bindings. Going from file-level → full CRDT later changes only how text bytes merge; the binding / provenance / review model never moves. The rejected line-number approach is precisely what couldn't have done this — positions are unstable the moment two people type at once.

7Against your priorities

Trust → UX → LLM-native → scale-as-a-constraint.
PriorityHow clean-file + lock-file serves it
1 · TrustBound values are token-anchored, so they can't silently drift. The one hard case — prose claims you can't tokenize — fuzzy-matches its anchor and fails loudly ("re-link?"), never silently mis-attributes. The file is always a real, readable file.
2 · UXRendered view is clean Word/Docs-grade prose; raw view shows honest, legible tokens; provenance is the blue gutter dot, pending is amber, applied is green — the designed grammar.
3 · LLM{{source.field}} actively tells an agent "this is derived — don't rewrite it," which is more legible than a hardcoded number. Skills, connectors, and chat all operate on near-pure Markdown.
4 · ScaleLock files are per-document and content-addressed by source + hash. Millions of docs = millions of small independent records, no global state. Bindings resolve and fan out independently.
